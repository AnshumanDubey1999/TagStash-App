## đź“‹ Q&A and Assumptions
* **Q**: What triggers the release workflow?
  * **A**: A manual trigger (`workflow_dispatch`) from the GitHub Actions tab.
* **Q**: What inputs are provided to the trigger?
  * **A**: A dropdown selection for the `release_type` with options: `patch` (default), `minor`, and `major`.
* **Q**: How is the APK signed?
  * **A**: For now, the `release` build type in `app/build.gradle.kts` will be configured to use the debug signing config. This produces a signed release APK that is immediately installable on test devices.
* **Q**: How are versions bumped?
  * **A**: Bumps `app.version.code` by +1. Bumps `app.version.name` based on the selected release type:
    * `patch`: Bumps the third digit (e.g. `0.1.0` -> `0.1.1`).
    * `minor`: Bumps the second digit and resets the third (e.g. `0.1.5` -> `0.2.0`).
    * `major`: Bumps the first digit and resets the second and third (e.g. `0.5.2` -> `1.0.0`).
* **Q**: What protections are in place for duplicate releases?
  * **A**: The workflow will fetch existing git tags and verify if the tag `v<new_version>` already exists. If it does, the pipeline aborts with an error before making any changes.
* **Q**: What are the release notes?
  * **A**: Automatically generated by GitHub using the delta of commits since the last release.

## đź“‹ Feature Requirements
1. **Gradle Build Configuration**:
   * Update the `release` build type block in `app/build.gradle.kts` to specify `signingConfig = signingConfigs.getByName("debug")` so the release APK is signed and installable.

2. **GitHub Actions Release Workflow**:
   * Create `.github/workflows/release.yml`.
   * Configure manual input:
     ```yaml
     on:
       workflow_dispatch:
         inputs:
           release_type:
             description: 'Release Type'
             required: true
             default: 'patch'
             type: choice
             options:
               - patch
               - minor
               - major
     ```
   * Workflow Steps:
     * Checkout codebase with full history (to fetch tags and push updates).
     * Read `app.version.code` and `app.version.name` from `gradle.properties`.
     * Calculate the bumped `versionCode` and `versionName`.
     * Check if git tag `v<new_version>` already exists. Abort if it does.
     * Update `gradle.properties` with the new version code and version name.
     * Commit `gradle.properties` and push back to `main`.
     * Create and push a git tag `v<new_version>`.
     * Set up Java JDK 21 (Temurin) and Gradle caching.
     * Build release APK using `./gradlew assembleRelease`.
     * Create a GitHub Release using `softprops/action-gh-release@v2`, attaching the generated APK from `app/build/outputs/apk/release/app-release.apk` (or similar path depending on flavor) as a release asset.
